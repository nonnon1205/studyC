# UT レビュー

指定されたテストファイルと対象ソースファイルをレビューし、以下の観点で指摘を出してください。
指摘は「不正（NG）」「推奨（WARNING）」「質問（Q）」に分類してください。

## 0. 事前トレース（レビュー開始前に必ず実施）

レビューの前に以下の2つの表を埋めてから各観点のチェックに進むこと。
表が埋まっていない状態で指摘を出してはいけない。

### 関数 × パラメータ トレース表

対象ファイルの全公開関数について、各 `@param` を行として列挙し、
「正常値テストあり / 異常値（NULL等）テストあり / 未テスト」を記入する。

| 関数名 | パラメータ | 正常値 | 異常値 | 備考 |
|---|---|---|---|---|
| （全公開関数を列挙） | （全 @param を列挙） | ○/✗ | ○/✗ | |

### 分岐 × テスト トレース表

ソース実装の全 `if` / `switch` 分岐を行として列挙し、
「真側テストあり / 偽側テストあり」を記入する。
比較演算子（`>` / `<` / `>=` / `<=`）を含む条件は「境界ピッタリ」列も記入する。

| 箇所（関数:条件式） | 真側 | 偽側 | 境界ピッタリ | 備考 |
|---|---|---|---|---|
| （全分岐を列挙） | ○/✗ | ○/✗ | ○/✗/N/A | |

---

## 1. ブランチカバレッジ（ホワイトボックス）

トレース表の✗をもとに以下を確認する。

- `if` / `else` の全分岐が通過するテストケースが存在するか
- `switch` の全 `case`（`default` 含む）を通過するテストケースが存在するか
- 比較演算子（`>`/`<`/`>=`/`<=`）に対して「境界の1つ下・ピッタリ・1つ上」の3点が揃っているか
- `uint8_t` 等の符号なし型に対して負方向の境界が考慮されているか（暗黙変換の理解）

## 2. ブラックボックス観点（Doxygen をテストベースとする）

トレース表の✗をもとに以下を確認する。

- `@param` の各引数（NULL 許容かどうかを含む）が**パラメータ単位で独立して**テストされているか
- 複数の NULL 許容パラメータがある場合、各引数を個別にテストしているか
- `@return` に記載された戻り値のパターンが全てテストされているか
- Doxygen に記載があるが実装・テストが存在しない仕様がないか（未実装の検出）
- Doxygen に記載がないが実装されている振る舞いがないか（仕様化漏れの検出）

## 3. 静的テスト観点

テストコード自体の品質を確認する。

- `setUp` / `tearDown` が適切に使われているか（テスト間の独立性）
- テスト関数名が「何をテストしているか」を表しているか
- `TEST_ASSERT_EQUAL` の引数順序が `(expected, actual)` になっているか
- 重複したテストケースがないか
- テストが実装の詳細に依存しすぎていないか（壊れやすいテストの検出）

## 4. KPI 確認

- ブランチカバレッジ：トレース表に✗が残っていないか（目標 100%）
- テスト成功率：全テストが PASS しているか（目標 100%）
- 関数カバレッジ：対象ファイルの全公開関数がトレース表に列挙されているか

## 出力形式

まずセクション0のトレース表を出力する。
その後、各指摘を以下の形式で出力する。

```
[不正] <テスト関数名または対象箇所> — <理由>
[推奨] <テスト関数名または対象箇所> — <理由>
[質問] <テスト関数名または対象箇所> — <理由>
```

最後に KPI の充足状況をまとめる。
