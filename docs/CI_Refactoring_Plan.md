# CI構築およびリファクタリング推進計画

## 目的
システムの境界値バグ（SIGPIPEクラッシュ等）やコードの粗（マジックナンバー散在等）を修正するにあたり、修正によるデグレを防ぐための自動テスト（CI）環境を先行して構築する。
テスト駆動開発（TDD）のサイクルを回すことで、安全かつ確実なシステム堅牢化を実現する。

## 運用ルール
今後は「コードを修正する前に、その挙動を保証（またはバグを再現）するテストが存在するか」を意識し、テストと実装をセットで進めること。

## 今後のロードマップと決定理由
現在の非常にクリーンな状態を維持・拡張するため、以下の順番で着手する。

1. **【Phase 4】エッジケーステストの自動化実装** (`EdgeCase_TestCases.md` に基づく)
   - **理由:** 新機能を追加してシステムが複雑化する前に、システムの「外壁（外部入力の境界）」に対する堅牢性を証明するため。ASanと組み合わせることで、最速かつ最大の範囲で実質的な単体テストとして機能させる。
2. **【Phase 5】単体テスト (Unit Test) 環境の構築と既存ロジックのテスト**
   - **理由:** エッジケースで外壁を固めた後、I/Oに依存しない「純粋な内部ロジック（`event_handler`等）」が仕様通りに動いているかを関数レベルできめ細かく保証するため。この内部の保証がないまま新機能（メトリクス等）を乗せると、バグの切り分けが困難になるため。
4. **【Phase 6】テストカバレッジ計測（gcov / lcov）の導入**
   - **理由:** 単体テスト構築（Phase 5）の段階でテスト用Makefileを整備するタイミングに合わせて導入する。E2Eテスト（外壁）ではどうしても網羅しきれない「内部ロジックの未通過ルート」を可視化するレーダーとして活用し、効率的な単体テストの実装を促進するため。
3. **【Phase 7】MgmtCtl 新機能（`GET_METRICS` / `RESET_METRICS` 等）の実装**
   - **理由:** 外壁（エッジケース）と内装（単体テスト）が完璧に整備された最強の土台の上に、満を持して新機能を追加する。新機能の開発自体も、構築した単体テスト環境を利用したTDD（テスト駆動開発）で行うことができるため。
