# CI構築およびリファクタリング推進計画

## 目的
システムの境界値バグ（SIGPIPEクラッシュ等）やコードの粗（マジックナンバー散在等）を修正するにあたり、修正によるデグレを防ぐための自動テスト（CI）環境を先行して構築する。
テスト駆動開発（TDD）のサイクルを回すことで、安全かつ確実なシステム堅牢化を実現する。

## フェーズと手順

### Phase 1: 自動テストの土台作り (E2Eテスト)
- システム全体を貫通する結合テストスクリプト（Python / `pytest` 等）を作成する。
- Collector(UDP受信) -> Router(SHM/MQ) -> Viewer(TCP送信) の一連のデータフローが正常に動作することを外部から検証できるようにする。
- Makefile に `make test` などのテスト実行用ターゲットを追加する。

### Phase 2: テスト駆動でのクリティカルな粗の修正
Phase 1で作ったテストを回しながら、以下の修正を安全に行う。
1. **ネットワーク異常系対策**: RouterプロセスでのTCP送信時における `SIGPIPE` クラッシュ対策（`MSG_NOSIGNAL`等の適用）。
2. **マジックナンバーの排除**: `7777`, `8888`, バッファサイズ等を `msg_common.h` などの共通ヘッダに集約し、モジュール間の契約をコード上で明示する。

### Phase 3: CIパイプラインの自動化
- GitHub Actions などの CI ワークフローファイルを作成する。
- push 時に以下のパイプラインが自動で走るようにする。
  1. ビルド検証 (`make all`, `make debug`)
  2. 静的解析 (`cppcheck`, `clang-tidy`)
  3. 結合テスト (Phase 1 のスクリプト)

### Phase 4: その他の高度な改善（次期課題）
- エッジケース（0バイトUDP、SHM異常値）のテストケース追加と対応。
- MgmtCtl（管理コマンド）の統合完了。
- リソースリーク検査（Valgrindの導入）等。

---
**運用ルール:**
今後は「コードを修正する前に、その挙動を保証（またはバグを再現）するテストが存在するか」を意識し、テストと実装をセットで進めること。
