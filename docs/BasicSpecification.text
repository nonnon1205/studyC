# 基本設計書

## 1. 概要
本システムは、複数のプロセスが連携してデータストリームを処理する非同期イベント駆動型のアプリケーションです。主なプロセスとして、データの中継・処理を担う TestMsgRcv と、最終的なデータを表示する TestUDPKill が存在します。

システム全体の特徴は以下の通りです。

- **マルチプロセス/マルチスレッドアーキテクチャ:** 各プロセスは特定の責務を持ち、内部ではI/O処理やシグナルハンドリングなどを個別のスレッドに分離しています。
- **イベント集約:** TestMsgRcv では、UDP、OSシグナル、プロセス間通信(IPC)など、性質の異なるイベントをすべてメッセージキューに正規化して集約し、メインスレッドで一元的に処理します。
- **プロセス間通信(IPC):** データ本体の共有には共有メモリ(SHM)を、更新通知にはSystem V メッセージキューを使用し、効率的なデータ交換を実現しています。高スループットデータはTCPソケットでプロセス間を転送されます。
- **外部管理インターフェース:** UNIXドメインソケットを介して、実行中の各プロセスに対して外部からコマンド（ログレベル変更、メトリクス取得、設定変更など）を送信できる、動的な管理プレーンを備えています。
- **堅牢な終了処理:** シグナルを安全に処理し、各スレッドがリソースを確実に解放してから終了する、グレースフルシャットダウンの仕組みが実装されています。

## 2. システムアーキテクチャ

### 2.1. コンポーネント構成
本システムは、主に以下のコンポーネントから構成されます。

```
┌───────────────────┐      ┌───────────────────┐      ┌───────────────────┐
│ PollIO (Producer) │      │   TestMsgRcv      │      │   TestUDPKill     │
│ (※想定)          │      │ (Router/Processor)│      │     (Viewer)      │
└───────────────────┘      └───────────────────┘      └───────────────────┘
         │                         │                         │
         │ 1. SHMにデータ書込        │                         │
         ├───────────────────►     │                         │
         │                         │                         │
         │ 2. MQで更新通知         │                         │
         ├───────────────────►     │                         │
         │                         │ 3. MQで通知受信         │
         │                         │    SHMからデータ読込      │
         │                         │                         │
         │                         │ 4. TCPでデータ転送      │
         │                         ├───────────────────►     │ 5. TCPでデータ受信
         │                         │                         │    コンソールに表示
         │                         │                         │
┌───────────────────┐      ┌───────────────────┐      ┌───────────────────┐
│  Admin Client     │      │      (共通)       │      │      (共通)       │
└───────────────────┘      └───────────────────┘      └───────────────────┘
         │                         │                         │
         │ 6. UNIXドメインソケット │                         │
         │    で管理コマンド送信   │                         │
         ├───────────----──►     │                         │
         │                         │                         │
```

### 2.2. プロセス間通信 (IPC)
- **共有メモリ (SHM) + メッセージキュー (MQ):** 高スループットなデータ交換に使用されます。データ生産者(例: PollIO)がSHMに実データを書き込み、軽量なメッセージキューでデータ受信者(TestMsgRcv)に更新を通知します。これにより、受信側はポーリングではなく効率的なブロッキング待機が可能です。
- **TCPソケット:** TestMsgRcv から TestUDPKill への、整形済みデータのストリーミング転送に使用されます。
- **UNIXドメインソケット:** 各プロセスに実装された管理インターフェースへのアクセスポイントとして機能します。外部の管理クライアントが接続し、コマンドを送受信します。

## 3. 主要コンポーネント詳細

### 3.1. TestMsgRcv プロセス
データソースからのイベントを集約し、TestUDPKill へ中継するハブとしての役割を担います。

- **責務:**
  - 複数の非同期イベント（UDP、シグナル、IPC）をメッセージキューで一元管理。
  - 共有メモリの更新通知を受け、データを読み出す。
  - 読み出したデータをTCPソケット経由で TestUDPKill に転送する。
- **内部スレッド:**
  - **main:** メインループ。メッセージキュー(mainmsqid)から正規化されたイベントを msgrcv で受信し、イベント種別に応じて処理を振り分けます。
  - **signal_worker:** sigwait を用いて SIGINT, SIGTERM を安全に捕捉し、メインスレッドへイベントとして通知します。
  - **router_worker:** 共有メモリの更新通知用メッセージキュー(ipc_msqid)を監視。通知を受け取ると shm_api_read でデータを読み出し、TCP接続を通じて TestUDPKill へ転送します。

### 3.2. TestUDPKill プロセス
TestMsgRcv から転送されたデータを最終的に受信し、ユーザーに提示する役割を担います。

- **責務:**
  - TestMsgRcv からのTCP接続を待ち受ける。
  - 受信したデータを標準出力に表示する。
- **内部スレッド:**
  - **main:** 全スレッドの起動と、グレースフルシャットダウンの管理を行います。
  - **tcp_worker:** poll を使用してTCPソケットとシャットダウン通知用のパイプを監視。TestMsgRcv からの接続を accept し、データを受信・表示します。
  - **signal_worker:** sigwait を用いてOSシグナルを捕捉し、メインスレッドの終了シーケンスを起動させます。
  - **ipc_worker:** 外部からのIPCメッセージを監視します。

### 3.3. 共通管理ライブラリ (Mgmt/)
各プロセスに組み込まれ、統一された管理機能を提供します。

- **mgmt_socket:** UNIXドメインソケット(SOCK_DGRAM)のサーバーとして機能します。管理クライアントからのコマンドリクエスト(MgmtCommandRequest)を受信し、レスポンス(MgmtCommandResponse)を返します。
- **event_handler:** コマンドのディスパッチャです。handler_register によってモジュール名とコマンド種別をキーとして、処理関数（ハンドラ）を動的に登録できます。mgmt_socket は受信したコマンドを handler_dispatch に渡し、適切なハンドラを呼び出します。
- **mgmt_protocol:** 管理コマンドのデータ構造、コマンド種別(MGMT_CMD_*)、結果コード(MGMT_RESULT_*)などを定義します。
- **unified_logger:** スレッドセーフな高機能ロガーです。ログレベル設定、syslog/ファイル出力、コンテキストタグの付与などの機能を持ちます。
- **metrics_collector:** レイテンシ、処理数、エラー数などのパフォーマンスメトリクスを収集・管理します。スナップショットをJSON形式でシリアライズする機能も備えています。

## 4. 主要なデータフロー

### 4.1. データ転送フロー
1. データ生産者（PollIOなど）が共有メモリにデータを書き込む。
2. 生産者はメッセージキュー（SYSTEM_IPC_KEY）に更新通知メッセージを送信する。
3. TestMsgRcvのrouter_workerスレッドが、msgrcvで通知をブロック待機して受信する。
4. router_workerはshm_api_readを呼び出し、共有メモリからデータを読み取る。
5. router_workerは確立済みのTCPソケットを使い、読み取ったデータをTestUDPKillへsendする。
6. TestUDPKillのtcp_workerスレッドが、pollでデータ到着を検知し、recvで受信する。
7. tcp_workerは受信データをコンソールに出力する。

### 4.2. 管理コマンドフロー
1. 外部の管理クライアントが、対象プロセスのUNIXドメインソケット（MGMT_SOCKET_PATH）にMgmtCommandRequestを送信する。
2. 対象プロセスのmgmt_workerスレッドがrecvfromでリクエストを受信する。
3. handler_dispatch関数が呼び出され、リクエスト内のtarget_moduleとcmd_typeに一致するハンドラをレジストリから検索する。
4. 見つかったハンドラ関数が、リクエスト内容を引数として実行される。
5. ハンドラは処理結果をMgmtCommandResponse構造体に書き込む。
6. mgmt_workerスレッドがsendtoでクライアントに応答を返す。

## 5. 終了処理 (Graceful Shutdown)
本システムは、Ctrl+C (SIGINT) や kill (SIGTERM) コマンドで安全に停止できるよう設計されています。

1. signal_workerスレッドがsigwaitでシグナルを捕捉する。
2. signal_workerは、グローバルな終了フラグ（g_keep_runningやshutdown_requested）をfalseに設定し、メインスレッドに通知する（メッセージキューや条件変数経由）。
3. メインスレッドは終了シーケンスを開始する。
4. select, poll, msgrcvなどでブロッキング待機しているワーカースレッドを起床させるため、特殊なメッセージ（"POISON_PILL"）を送信したり、監視対象のパイプに書き込みを行ったりする。
5. 各ワーカースレッドはループ条件（終了フラグ）が偽になったことでループを抜け、ソケットやファイルディスクリプタなどのリソースを解放して正常に終了する。
6. メインスレッドは、pthread_joinですべてのワーカースレッドの終了を待機する。
7. すべてのスレッドが合流した後、メインスレッドは残りのリソース（メッセージキューなど）を解放し、プロセスを終了する。
