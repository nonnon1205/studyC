# テスト戦略およびディレクトリ構成仕様書

## 1. テスト戦略の基本方針
本プロジェクトでは、システム全体のデグレを防ぎながらリファクタリングを安全に進めるため、「テストピラミッド」を逆順に構築するアプローチを採用します。

1. **Phase 1: システム結合テスト (E2Eテスト) の先行構築**
   - **目的:** 既存の動作を保証するための大きなセーフティネットを張る。
   - **ツール:** Python (`pytest`)
   - **対象:** プロセス間通信 (UDP -> SHM/MQ -> TCP) の貫通テスト、異常系パケットの入力、管理コマンド（`mgmtctl`）の動作確認。

2. **Phase 2: 単体テスト (Unitテスト) の段階的導入**
   - **目的:** リファクタリングによってモジュールの責務が整理された後、個々の関数ロジックをきめ細かく検証する。あわせて、**AddressSanitizer (ASan) を用いた徹底的なリソースリーク・メモリ破壊検査**を行う。
   - **ツール:** C言語向け軽量テストフレームワーク（**Unity** を想定）、**AddressSanitizer (ASan)**
   - **対象:** `Common/` (統合ロガー等)、`Mgmt/` (コマンド解析・JSONパース等)、`SHM/` などの共通ライブラリの純粋なロジック部分。
   - **CI連携:** テストビルド時に `-fsanitize=address` を付与し、既存のE2Eテストや単体テストの実行時にメモリリークや不正アクセスを自動検知してCIパイプラインでブロックする仕組みを構築する。

## 2. テストディレクトリ構成
プロジェクトルート直下に `tests/` ディレクトリを配置し、E2EテストとUnitテストを明確に分離します。

```text
studyC/
├── tests/                     <-- テスト関連ルート
│   ├── requirements.txt       <-- E2Eテスト用Pythonパッケージ定義 (pytest等)
│   │
│   ├── e2e/                   <-- システム結合(E2E)テスト (Python)
│   │   ├── conftest.py        <-- E2Eテストの共通設定・フィクスチャ (プロセス起動/停止)
│   │   ├── test_data_pipeline.py <-- データフローテスト
│   │   └── helpers/           <-- E2Eテスト用補助モジュール (UDPクライアント等)
│   │
│   └── unit/                  <-- 単体(Unit)テスト (C言語 / Unity等) [将来拡張用]
│       ├── Makefile           <-- Unitテスト用ビルドスクリプト
│       ├── test_shm_api.c     <-- APIの単体テスト
│       └── test_logger.c      <-- ロガーの単体テスト
```

## 3. テスト実行の取り決め (Contract)
- **標準エントリポイント:** プロジェクトルートでの `make test` コマンドにより、すべてのテスト（E2E、および将来的なUnit）が実行されること。
- **CI前提:** テストは GitHub Actions 等の CI 環境で、ヘッドレス（非対話型）で実行できる設計とすること。

## 4. 重要な仕様・規約
1. **リソースリークの確実な検出とクリーンアップ:**
   - テスト終了後、OS上に共有メモリやMQ、ソケットの残骸が残っていないかをテストフレームワーク（`conftest.py`）が検査する。
   - **残骸を発見した場合はテストを「失敗（リソースリーク）」として扱う。** その後、次回のテストのために強制的にクリーンアップを実行する。
2. **環境変数による環境の識別とオーバーライド:**
   - ポート番号やIPCキーなどのマジックナンバーは、テスト環境と本番環境が競合しないよう、環境変数（例: `TEST_UDP_PORT`）でオーバーライド可能な設計とする。（C言語側の実装はPhase 2で行う）
3. **テストカバレッジの計測:**
   - テストがコードのどの部分を通過したかを計測する仕組み（`gcov`/`lcov`）を将来的に導入し、CIパイプラインで可視化する。
